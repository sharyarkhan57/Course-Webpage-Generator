Developing the Course Site Generator Application

HW 5 - Implementation Stage II

In the second stage of our implementation we'll setup all of the file management. This means saving and loading application data as well as exporting course Web sites with properly working JavaScript code. Since we have not yet implemented all the UI interactions for entering data we'll have to artificially hard-code the creation of test data in order to test our file I/O. For this, let's use a separate set of test classes (i.e. drivers) to test our code. For these test classes, create a separate sub-package named test_bed within your application's home package where you'll keep them. So the test bed in HW 1 would have been put in tam.test_bed. The idea is that you'll prove your code for File I/O works independent of its use within the application's UI.

Requirements

In this assignment you must complete the following:

  • Data Saving - create a separate test class called TestSave in your test bed to test and demonstrate that complete app data saving works. Have your driver class hard-code the creation of all the necessary data, meaning good data for every UI control, including UI tables, and then send that data to be saved to your file manager such that it may write it to a file name called SiteSaveTest.json. You should be using a JSON format which has the advantage of being human readable. So, examine your output file to ensure it is in its proper format.

  • Unit Testing Loading - now create a series of JUnit tests that will use the hard-coded data written to SiteSaveTest.json as its expected type and the data loaded as its actual type. So, using JUnit, you should be verifying that all the data written to the JSON file loads properly. Note that the point of using JUnit is that it provides a consistent format for setting up tests and will compile stats for results.

  • UI Loading Integration - once you have implemented and tested saving and loading, integrate loading into your application's UI. Note that the user should now be able to press the load button and select the file to load. Then, the file's data should fully load into your user interface controls, meaning for all five screens.

  • Site Exporting - Now it's time to get exporting working. When the user clicks the Export button it should export the full site according to the currently loaded settings. This menas all HTML pages and directories that are part of the site template (like css, images, js, etc.). Note that you will have to consider at this time how to properly export data into the various .json files for the site pages, and then how to properly load the data vis JavaScript, so yes, in this assignment you're going to have to write a bit of JavaScript. Now, the important thing to note is everything that is to appear in a page that may be edited via the UI must be loaded by JavaScript into the page. Right? This means even things like the course name and number. It's important that you understand this before diving in because this will determine the formats of your .json files and how your JavaScript works. Note that you may use any JavaScript code you like from my existing courses like CSE 219, CSE 308, and CSE 380. Be careful with these sites, by the way, there are some slight differences versioning differences between the .json and JavaScript files you may find there.

Handin Instructions

When you are done, zip up and submit any projects where you changed code as a single zip file. Note that you should add your name as the author to any source file you provide.
